TASK 1: REVIEW & REFACTOR
========================================

Issues I found:

1. Most severe — Performance Killer: 
   All three GET routes fetch policies, then fire a separate agent query per policy inside a loop. For 100 policies that's 101 DB roundtrips. 

2. Unvalidated req.body passed directly to Prisma (mass assignment) in POST "/".
   A client can inject any field — including id, agentId, status, or any future sensitive field — with no sanitization or schema validation.

3. Missing try/catch in '/expiring' and '/:id' — any DB error becomes an unhandled promise rejection that can crash the Node process. 
   /:id uses findUniqueOrThrow with no catch, so a missing ID returns a raw 500 with Prisma internals exposed to the client instead of a proper 404.

4. If someone hits /policies/abc, Number("abc") returns NaN. Prisma receives { id: NaN } — behavior is undefined and varies by DB driver. 
   There's no guard that returns a 400 before the query even runs. 

5. new PrismaClient() instantiated per module, Each module that does this creates its own connection pool. 
   In a larger app with multiple route files, you end up with multiple pools competing for DB connections.

6. Repeated use of hard coded string - 'Unknown' appears 3 times and a named constant is better than a repeated magic string.

7. Const where: any = {}, use of "any" defeats TS entirely and considered bad practice.

What I fixed (in priority order) and why:

1. Issue: Performance Killer
   Why this was the priority: At scale, the N+1 problem degrades every single request to the list endpoints. 
   It's a structural flaw that gets worse with data growth and can make the service effectively unusable under load.

2. Issue: Unvalidated req.body passed directly to Prisma (mass assignment) in POST "/"
   Why this was the priority: Passing req.body directly to Prisma is a mass assignment risk — a client can set any field, 
   including sensitive ones like status or agentId. 
   This is especially dangerous with financial and insurance data where tampering can have legal and financial consequences. 
   Zod also enforces input validation (types, enums, required fields) that was missing entirely before.

3. Issue: Missing try/catch on /expiring and /:id can cause unhandled promise rejections that crash the Node process, 
   and exposes raw Prisma error internals to the client instead of clean HTTP responses (e.g. 500 instead of 404 for a missing record). 
   Ranked lower than N+1 and mass assignment because it only triggers on error paths rather than on every request, 
   and a process crash is recoverable via a process manager like PM2 — whereas N+1 degrades every single request and 
   mass assignment is a silent data integrity risk.

What I left unfixed and why:

1. Issue: /:id doesn't have type safty for 'id' and the correct error handling.
   Why I left it: Not critical as the 3 issues that where fixed, we still have the try / catch around.

2. Issue: Usage of "any" in TypeScript.
   Why I left it: Usage of any in TypeScript disables type checking on the where filter object, 
   meaning invalid field names or values won't be caught at compile time. 
   Left unfixed because it doesn't affect runtime behavior or business logic.

3. Repeated use of hard coded string - 'Unknown'.
   Why I left it: Left unfixed because the string is only used within this one file and never shared across modules, 
   so the risk of inconsistency is minimal. 
   Worth extracting to a constant if the codebase grows, but premature for the current scope.

TASK 2: BUILD A FEATURE
========================================

Assumptions I made:

1. There's a table in the db for agents.
2. In that table the agent id is unique.
3. The agent ID is known to the frontend at page load (e.g. from a session/auth token or URL param) — the endpoint requires it, 
   so the client must have it before making the call.
4. An unknown agentId returning empty results (rather than a 404) is acceptable — 
   there's no auth/access-control layer that needs to distinguish "agent doesn't exist" from "agent has no policies."   

Design choices:

API:

-  Added a new route prefix - agents.
   Rather than adding agent-scoped routes under the existing /policies prefix (e.g. /policies?agentId=...), a dedicated /agents router was introduced. 
   This reflects the resource hierarchy: an agent is considered in my view as a first-class entity (not at the current code but it's trivial for this system), 
   and their policies are a sub-resource of that entity — /agents/:agentId/policies reads naturally as "the policies belonging to this agent."

-  Nested resource route instead of a query param:
   Chose GET /agents/:agentId/policies instead of adding agentId as a filter to the existing GET /policies. 
   The agent is the primary context for this feature — a nested route makes that explicit and non-optional, 
   preventing accidental calls that return all policies with no agent scope.

-  No separate expiring endpoint:
   Decided against GET /agents/:agentId/expiring-policies. The task asks for one page that shows all policies and highlights the ones needing attention.
   A second endpoint would mean two calls returning the same data model with overlapping results.
   Instead, the single endpoint supports sortBy=endDate&sort=asc so expiring policies naturally surface at the top, 
   and the frontend highlights rows where endDate falls within the next 30 days — a simple date check on data already in the response.

-  Pagination and sorting as query params:
   Added size, offset, sortBy, sort. With ~200 clients per agent, returning everything at once is impractical. Offset-based pagination is straightforward and sufficient at this scale (no need for cursor-based).

-  Zod validation on query params:
   AgentPoliciesQuerySchema coerces size/offset from strings to numbers (query params are always strings), applies defaults, and whitelists sortBy to actual model fields — preventing injection of arbitrary Prisma orderBy keys. Consistent with how the existing POST /policies uses CreatePolicySchema.

Frontend:
-  Created these components - AgentsTable, PoliciesTable, Pagination (shared)

-  Extracted API calls to a separate service file (agents-service.ts)

-  Split policies into two tables, active and non-active policies. policies that are about to expire are highlighted in a warning color.

-  Use custom hook useGetPolicies to encapsulate logic.

-  Offset-based pagination with a configurable page size constant

-  TypeScript interfaces for all entities (Agent, Policy, etc.) with union types for status/type fields

-  Master-detail layout — selecting an agent loads its policies; state is held at the App root level and passed down

-  Local state only — no external state management library (Redux/Zustand); useState/useEffect throughout

What I'd do differently with more time:

1. Create a login page for an agent using email + password
2. Create an auth login route for agent whilst receiving jwt token and keeping it etc.
3. Add loader for a nicer feel for users.
4. Use components from a components library like "base-ui" or "radix-ui"
5. Use a Zustand store for some of the states
6. Use ReactQuery instead of using the methods in the agents-service directly
7. Improve UI look and feel
